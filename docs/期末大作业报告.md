## 摘要

本项目构建了一套 **AI 驱动的 FPGA 开发工作流**，以 Claude Code 作为核心 AI 引擎，通过系统级 Prompt 工程（CLAUDE.md）与自动化工具链（sim.sh），实现了从需求描述到 RTL 代码生成、Testbench 编写、仿真验证、波形调试的完整自动化开发闭环。作为该工作流的验证案例，项目成功交付了一款 **LEG 8位处理器**——包含 27 条指令、6 个通用寄存器、256 字节数据存储器及硬件栈，并配套构建了 **Web 可视化 IDE**（含 JavaScript 汇编器与周期精确模拟器）。全部 29 个 RTL 模块的 114 项测试用例通过 Vivado 仿真验证，验证了 AI 辅助硬件设计的可行性与有效性。

**关键词**: AI 辅助编程；FPGA 开发；Verilog；大语言模型；处理器设计；自动化工作流；Claude Code

## 一、项目背景与问题分析

### 1.1 FPGA 开发的教学困境

FPGA（现场可编程门阵列）开发是计算机专业硬件类课程的核心内容，但存在显著的入门壁垒：

1. **工具链复杂**: Vivado 等 EDA 工具安装配置繁琐，从新建工程到第一个仿真通过往往需要数小时；
2. **调试困难**: 硬件描述语言（HDL）的并发执行模型异于软件编程，时序问题难以定位，波形分析门槛高；
3. **反馈周期长**: 综合-布局布线-生成比特流-上板验证的循环以小时计，挫伤学习积极性；
4. **缺乏结构化指导**: 学生独自面对空白编辑器和数十万页的 EDA 文档，缺乏即时的编码规范提示和错误诊断。

### 1.2 AI 辅助硬件设计的机遇

大语言模型（LLM）在代码生成、错误诊断和自动化测试领域展现了卓越能力。然而，现有 AI 编程助手（如 GitHub Copilot、Cursor）主要面向软件编程，缺乏对硬件描述语言工作流的深度适配。本项目探索的核心问题是：**能否通过系统级 Prompt 工程和工具链集成，让 LLM 成为 FPGA 开发的"结对编程伙伴"，完整驱动 RTL 编写→仿真验证→调试修复的开发闭环？**

### 1.3 项目目标

- 构建一套可复用的 AI 驱动 FPGA 开发工作流（CLAUDE.md + 仿真脚本套件）
- 以 LEG 处理器作为非平凡的验证案例，证明工作流能够产出复杂数字系统
- 提供可视化交互环境，降低处理器原理学习门槛
- 验证该方法论在课程教学中的应用价值

## 二、模型选择与技术路线

### 2.1 模型选型

选用 **Claude（Anthropic）** 作为核心 AI 引擎，通过 **Claude Code CLI** 实现本地化开发集成。选择理由如下：

| 考量维度 | Claude Code 优势 |
|---------|-----------------|
| 代码生成能力 | 对 Verilog/SystemVerilog 语法有深入理解，可生成符合工业规范的综合代码 |
| 长上下文支持 | 100K+ token 上下文窗口，可同时加载完整 ISA 文档、编码规范和多模块源码 |
| 工具调用（Function Calling） | 原生支持 Bash 命令执行，可直接驱动 Vivado 编译和仿真 |
| 系统级 Prompt | 支持 CLAUDE.md 项目管理文件，实现持久化的编码规范和调试流程注入 |
| 自主调试循环 | 能解析仿真日志，自动判断语法错误/连接错误/逻辑错误并迭代修复 |

### 2.2 技术路线

项目采用 **"Prompt 驱动 + 工具链编排 + 迭代调试"** 三层架构：

```
需求描述
    │
    ▼
┌──────────────────────────────────────┐
│  CLAUDE.md（系统级 Prompt）           │
│  · Verilog 编码规范                   │
│  · Testbench 编写要求                 │
│  · 调试流程指导                       │
│  · 模块模板与命名约定                 │
└──────────────────────────────────────┘
    │
    ▼
AI 编写 RTL ──→ AI 编写 TB ──→ sim.sh 仿真
    ▲                                │
    │                                ▼
    └────── AI 调试修复 ←────── 检查日志/波形
              (最多5轮迭代)          │
                              ┌─────┴─────┐
                              │  通过？     │
                              └─────┬─────┘
                                    ▼
                                  完成
```

- **AI 层**: Claude Code 负责代码生成、错误诊断和修复决策
- **工具层**: sim.sh 编排 Vivado 2025.1 工具链（xvlog 语法分析 → xelab 设计细化 → xsim 仿真执行）
- **验证层**: parse_sim_log.py 解析仿真日志提取结构化错误，波形导出支持时序深度分析
- **展示层**: Web GUI 提供汇编编程、模拟执行和状态可视化的交互界面

## 三、系统设计与功能实现

### 3.1 总体架构

系统由四个层次组成：

| 层次 | 组成 | 功能 |
|------|------|------|
| **AI 工作流层** | CLAUDE.md + Claude Code | 系统级 Prompt，定义编码规范、TB 模板和调试策略 |
| **工具链层** | sim.sh / run_sim.tcl / wave_view.tcl / parse_sim_log.py | 编排 Vivado 仿真全流程，提供日志解析与波形分析 |
| **LEG 处理器层** | leg_lib（19个基础模块）+ leg_core（8个核心模块） | 完整的 8 位微处理器 RTL 实现 |
| **可视化展示层** | leg_gui.html（~1200行单文件Web应用） | 汇编编辑器 + JavaScript 汇编器 + 周期精确模拟器 + 状态面板 |

### 3.2 LEG 处理器微架构

LEG 处理器采用经典的五级数据通路结构：

```
PC → 取指(Program ROM) → 译码(Decoder) → 寄存器读(RegFile) → ALU运算 → 写回
                                                   ↓                ↓
                                              Stack(CALL/RET)   DMEM(LD/ST)
```

**核心模块说明**:

| 模块 | 功能 | 关键特性 |
|------|------|---------|
| `leg_pc` | 程序计数器 | 支持绝对跳转（JMP）、条件跳转（JZ/JNZ/JC）和 HALT 自锁 |
| `leg_decoder` | 指令译码器 | 解析 32 位指令，输出 20 个控制信号，涵盖 27 条指令 |
| `leg_regfile` | 寄存器文件 | 6×8位通用寄存器 + PC/IN 读取，双端口读、单端口写 |
| `leg_alu` | 算术逻辑单元 | 8 种运算（ADD/SUB/AND/OR/NOT/XOR/SHL/SHR），Z/C 标志输出 |
| `leg_stack` | 硬件栈 | 256 深度 × 8 位，支持 CALL/RET 嵌套调用 |
| `leg_dmem` | 数据存储器 | 256×8 位，LD/ST 指令访问 |

### 3.3 LEG 指令集架构

LEG 处理器支持 27 条指令，分为六大类：

| 类别 | 指令 | 说明 |
|------|------|------|
| 算术运算 | ADD, SUB, ADDI, SUBI, CMPI | 加减运算与比较 |
| 逻辑运算 | AND, OR, NOT, XOR, ANDI, ORI, XORI | 按位逻辑操作 |
| 移位运算 | SHL, SHR, SHLI, SHRI | 逻辑移位 |
| 数据传送 | LD, ST, LDI, MOV | 访存与寄存器传送 |
| 控制流 | JMP, JZ, JNZ, JC, CALL, RET, HALT | 跳转与子程序调用 |
| 栈与I/O | PUSH, POP, IN, OUT, OUTI | 栈操作与输入输出 |

每条指令固定 4 字节（32 位），小端序存储。Byte0 为操作码，Byte1/Byte2 编码操作数与目标寄存器，Byte3 的 bit[3] 控制 ALU 标志更新。

### 3.4 Web 可视化 IDE

`leg_gui.html` 是一个零依赖的单文件 Web 应用，功能包括：

- **汇编编辑器**: 支持行号、LEG 语法高亮和标签（Label）跳转目标
- **JavaScript 汇编器**: 将 LEG 汇编源码编译为 4 字节机器码，支持 .ORG/.DB/.DW 伪指令和标签解析
- **JavaScript 周期精确模拟器**: 完整复现 Verilog `leg_core` 的行为，包括 6 个寄存器、256 字节 RAM、256 深度栈、Z/C 标志和 HALT 检测
- **状态面板**: 实时显示 R0-R5、PC、SP、Z/C 标志和当前执行指令
- **Demo 程序**: 内置 Fibonacci 数列、算术全运算、GCD 辗转相减法三个示例程序

### 3.5 仿真验证平台

`sim.sh` 是仿真流程的自动化驱动脚本，支持以下命令：

```bash
./scripts/sim.sh sim    <module>   # 完整仿真 (xvlog → xelab → xsim)
./scripts/sim.sh wave   <module>   # 打开 Vivado 波形 GUI
./scripts/sim.sh export  <module>   # 导出 VCD 文本波形
./scripts/sim.sh parse   <module>   # 解析仿真日志提取错误
./scripts/sim.sh clean   <module>   # 清理仿真产物
```

AI 调试循环中，Claude Code 自动调用这些命令，根据输出结果判断仿真状态并决定下一步行动。

## 四、Prompt 工程与智能体方案

### 4.1 CLAUDE.md — 系统级 Prompt 设计

`CLAUDE.md` 是本项目的核心 Prompt 工程产物，它在 Claude Code 每次启动时自动注入系统上下文，内容涵盖：

1. **Verilog 编码规范**: `i_`/`o_`/`w_` 端口与连线命名约定（如 `i_enable`, `o_valid`, `w_ready`），时钟(`clk`)/复位(`rst_n`低有效)统一命名，端口顺序（时钟→复位→控制→数据入→数据出→状态）规范
2. **Testbench 编写标准**: 必须包含 `$dumpfile`/`$dumpvars` 波形导出、自检断言（PASS/FAIL 标记 + `$error` + `$display`）、汇总报告（通过/失败数量统计）和 task 封装的复用测试操作
3. **仿真工作流**: sim.sh 五个子命令的用法和参数约定
4. **AI 调试循环规则**: 编译失败（xvlog/xelab）→查编译日志→修复语法/连接；仿真失败（$error/$fatal）→ parse 日志→定位根因；波形异常→export VCD→分析时序。最多迭代 5 轮
5. **文件创建规则**: RTL 模块命名、TB 命名匹配、模块模板引用路径

### 4.2 工具链作为智能体"手"与"眼"

Claude Code 的原生 `Bash` 工具使其能够直接调用 Vivado 命令行工具：

- **"手"**: 执行 sim.sh 驱动仿真
- **"眼"**: 读取仿真日志（`.log`）、VCD 波形（`.vcd`）和 parse_sim_log.py 的结构化输出
- **"脑"**: 基于 CLAUDE.md 中编码规范和调试流程进行推理决策

这构成了一个完整的 **感知-推理-行动** 智能体循环：AI 编写代码 → 执行仿真 → 观察结果 → 定位问题 → 修复代码 → 重新验证。

### 4.3 LEG ISA 知识库

`docs/leg_isa.md` 是结构化的指令集规格文档，包含：
- 27 条指令的完整操作码映射表
- 每条指令的汇编语法、机器码编码格式和功能描述
- 寄存器地址映射和 I/O 端口说明

该文档在 AI 编写汇编程序（如 Demo 程序）和构建 JavaScript 模拟器时，作为参考知识库确保生成代码与 ISA 规格一致。

### 5.3 典型调试案例：GCD 死循环问题

GCD Demo 程序的调试过程充分体现了 AI 辅助调试的价值：

**问题现象**: GCD 程序在 Web GUI 中执行时出现死循环，程序在少数几条指令间反复跳转，无法到达 HALT。

**调试过程**:
1. 向 Claude Code 提供浏览器控制台输出的 30 步执行跟踪（含 PC、寄存器值、标志位状态）
2. Claude Code 识别出异常：第 1 步执行 `LDI R1, 18` 后，R1 仍为 0，而 R0 被错误地覆写为 18
3. 进一步追踪发现根因：JavaScript 模拟器中 `executeOneInstruction()` 函数使用 `rd = byte3 & 0x07` 作为所有指令的目标寄存器默认值
4. 但 LEG ISA 规定 LDI 指令的目标寄存器编码在 byte2（`i_inst[23:16]`），而非 byte3。类似地，LD 和 IN 指令也使用不同的字节位置
5. 修复方案：在 LDI（0x12）、LD（0x10）、IN（0x50）的 switch case 内显式指定 `rd = byte2 & 0x07`（IN 为 `rd = byte1 & 0x07`）
6. 修复后 GCD 程序正常输出正确结果 0x06

**启示**: 该 bug 极难通过人工代码审查发现——指令编码中目标寄存器位置的不一致性是一种"静默"语义错误。AI 通过系统化地追踪每条指令对寄存器状态的影响，快速定位了根因。这证明了 LLM 在硬件调试中具备超越简单语法检查的深层语义分析能力。

## 六、应用价值与不足

### 6.1 应用价值

1. **降低 FPGA 开发门槛**: 学生使用本项目工作流后，只需用自然语言描述设计需求，AI 即可生成符合编码规范的 RTL 代码和自检测试平台，将"从零到第一个仿真通过"的时间从数小时压缩到分钟级
2. **教学工具价值**: LEG 处理器及其 Web IDE 可作为《计算机组成原理》《数字系统设计》课程的配套实验平台，学生无需安装 Vivado 即可学习汇编编程和 CPU 微架构原理
3. **方法论可迁移**: CLAUDE.md + 脚本工具链的模式不限于 FPGA——任何需要特定编码规范和工具链的领域（嵌入式开发、芯片验证等）均可复用此方案
4. **AI 辅助硬件设计的实证**: 本项目是"用 AI 做硬件设计"的一次完整实践，从 ISA 定义到 RTL 实现到测试验证的全流程均由 AI 深度参与

### 6.2 不足与改进方向

1. **模型依赖**: 目前仅验证了 Claude Code 的效果，尚未适配 DeepSeek、通义千问等国产模型。后续可增加多模型对比评测
2. **离线能力不足**: 依赖商业 LLM 的云端 API，无法在完全离线环境中运行。可探索 Ollama + DeepSeek-Coder 的本地部署方案
3. **仿真深度限制**: 当前的自动化验证止于行为级仿真（RTL 功能验证），未覆盖综合后的门级仿真和时序收敛分析
4. **GUI AI 集成缺失**: Web IDE 目前是静态的汇编器和模拟器，未接入 LLM 实现"自然语言→LEG 汇编"的实时交互生成
5. **测试用例自动生成**: 当前 114 个测试用例多为手工或 AI 辅助编写，尚未实现基于覆盖率驱动的自动测试生成

### 旁白文稿
> 整套工作流的核心是 CLAUDE.md 文件——这是一个系统级的 Prompt 工程产物。它定义了 Verilog 编码规范、Testbench 编写标准、以及完整的 AI 调试循环策略。当 Claude Code 启动时，这个文件会被自动加载，使 AI 拥有 FPGA 开发领域的专业上下文。
>
> sim.sh 是连接 AI 和 Vivado 工具链的桥梁。Claude Code 通过 Bash 工具调用它，依次执行 xvlog 语法分析、xelab 设计细化、xsim 仿真执行。这是 leg_alu 模块的一次仿真，可以看到三阶段全部 PASS。

> AI 调试循环的策略是：编译失败时检查 xvlog/xelab 日志；仿真失败时 parse 日志定位断言错误；波形异常时导出 VCD 进行时序分析。最多迭代 5 轮，5 轮后仍未解决则请求人工介入。

### 旁白文稿
> 作为工作流的验证案例，我们设计了一款 LEG 8 位处理器。它采用经典五级数据通路：取指、译码、寄存器读、ALU 运算、写回。
>
> 核心模块包括：程序计数器 leg_pc、指令译码器 leg_decoder——它从 32 位指令中解析出 20 个控制信号，寄存器文件、支持 8 种运算的 ALU、256 深度的硬件栈、以及数据存储器。
>
> LEG 指令集共 27 条指令，涵盖算术运算、逻辑运算、移位、数据传送、控制流和栈操作六大类。每条指令固定 4 字节，小端序存储。正是因为我们编写了这份结构化的 ISA 文档，AI 才能在后续准确地生成汇编程序和 JavaScript 模拟器。

## 段落 6：GCD 调试案例（60秒）

### 画面
- 展示 GCD bug 相关的 Git 提交历史（`git log --oneline`）
- 展示浏览器 Console 的调试输出
- 展示修复后的代码 diff（LDI 的 switch case 修改）
- 如有时间，展示一张问题分析图

### 旁白文稿
> GCD 程序的调试是 AI 辅助调试价值的集中体现。最初 GCD 程序出现了死循环。我们要求 AI 在模拟器中加入逐步追踪日志，然后分析 Console 输出的 30 步执行轨迹。
>
> AI 发现了一个极难通过人工审查发现的 bug：LDI 指令在 JavaScript 模拟器中将目标寄存器错误地编码在了 byte3，而 ISA 规范定义它在 byte2。这导致 `LDI R1, 18` 错误地写入了 R0，R1 始终为 0，48 减 0 永远等于 48，程序死循环。
>
> 修复后程序正常输出 6。这个案例证明：AI 的系统化逐指令追踪能力，能够发现人工容易忽略的指令编码语义错误。这是传统调试手段难以做到的。

### 关键画面节点
- [00:00] git log 展示 5 个 GCD 相关提交
- [00:10] Console 日志展示寄存器追踪
- [00:30] 代码 diff：修复前 vs 修复后的 switch case
- [00:45] 修复后 GCD 输出正确结果
